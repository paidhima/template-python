# Python Project Template: Copilot Instructions

## Project Context
This is a Python project using:
- **Testing**: pytest with src/ layout (src/PROJECT_NAME/, tests/)
- **Type checking**: mypy in strict mode
- **Formatting**: black (line length 100)
- **Linting**: ruff with Google docstring style
- **Versioning**: semver traced in __version__ or pyproject.toml

## Development Workflow

### Required checks before submitting code
1. **Type check**: `mypy src/` (must pass with no errors)
2. **Format**: `black --check src/ tests/` (or auto-fix with `black src/ tests/`)
3. **Lint**: `ruff check src/ tests/` (must have no high-severity violations)
4. **Tests**: `pytest -xvs` (all tests must pass; new features require new tests)
5. **Coverage**: Run `pytest --cov=src/` (optional; ensure coverage is acceptable)

### Quick commands
- `python -m pytest tests/ -xvs` — Run tests with verbose output, stop on first failure
- `python -m mypy src/ --strict` — Full type check
- `python -m black src/ tests/docs/` — Format code
- `python -m ruff check src/ tests/ --fix` — Lint and auto-fix

## Branch Policy
- **main**: Only stable, tested code. No direct pushes; PR + approval required.
- **develop** (if used): Integration branch; PRs from feature branches.
- **Feature branches**: `feature/*, bugfix/*, refactor/*` based on issue or task.
- **Naming**: Use kebab-case with issue reference: `feature/issue-123-add-async-support`

## Documentation Requirements
1. **Module/class docstrings**: Google style with Args, Returns, Raises sections.
2. **Function signatures**: Type hints required; complex types go in docstrings.
3. **README.md**: Installation, quick start, core API by example, link to full docs.
4. **docs/ folder** (if complex): Markdown files for architecture, design, running tests.

Example docstring:
```python
def process_data(input_path: str, validate: bool = True) -> dict[str, Any]:
    """Process input file and return structured data.
    
    Args:
        input_path: Path to input JSON file.
        validate: If True, validate output schema. Defaults to True.
    
    Returns:
        Dictionary with keys 'success' and 'data'. Data is None if validation fails.
    
    Raises:
        FileNotFoundError: If input_path does not exist.
        ValueError: If file is not valid JSON or fails validation.
    """
```

## Dependencies Management
- **All dependencies**: Declared in pyproject.toml [project] dependencies or [project.optional-dependencies]
- **Dev dependencies**: [project.optional-dependencies] dev (pytest, mypy, black, ruff, etc.)
- **Lock files**: If needed (pip-tools, Poetry, uv), lock files committed to repo

## Security and Privacy
- **Secrets**: Never commit secrets; use environment variables or .env (in .gitignore).
- **Logging**: Avoid logging sensitive data (PII, tokens). Use structured logging if external sinks.
- **Data paths**: If handling file paths or user data, validate and sanitize inputs.

## Version Constraints
- **Python**: Target Python 3.11+. Call out breakpoints in CHANGELOG.
- **Dependencies**: Pin major versions; allow minor/patch. Migrate deprecated APIs promptly.

## Out-of-Scope Rules
- **No refactors** unless explicitly requested.
- **No restructuring** of existing tests without justification.
- **No major dependency upgrades** without discussion.
- **No type: ignore comments** without explanation.

## Completion Criteria
A task is done when:
- [ ] All changes pass `mypy src/ --strict` with zero errors
- [ ] All tests pass: `pytest tests/ -xvs`
- [ ] Code formatted: `black --check src/ tests/`
- [ ] Linting passes: `ruff check src/ tests/`
- [ ] New tests added for new logic
- [ ] Docstrings follow Google style; public API documented
- [ ] Commit message is clear and references issue (if applicable)
- [ ] Changes scoped to the stated task (no unrelated refactors)
